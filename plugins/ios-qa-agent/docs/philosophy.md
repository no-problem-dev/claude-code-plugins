# 設計思想とこだわり

## 根本的な問い

「人間がスマホをポチポチして動作確認する」という作業を、AI にどう委譲するか。

従来の XCUITest は「どうやって操作するか（How）」をコードで記述する。ボタンの ID を指定し、タップして、アサーションを書く。UI が変わればテストコードも書き直す。テストコードの保守が本来のテストの目的を上回る。

ios-qa-agent は発想を逆転させた。「何を確認したいか（What）」だけを自然言語で書く。「どうやって操作するか」は AI が自分で考える。UI が変わっても、確認したいことが変わらなければテストケースはそのまま使える。

## 3つの設計原則

### 1. 実行と判定の分離 — 確認バイアスの排除

これが最も重要な設計判断。

人間の QA でも、テストを実行した人と結果を判定する人を分けるのがベストプラクティス。自分でやったことは「うまくいったはず」と思いがちだから。

AI でも同じ問題が起きる。qa-runner が操作して報告し、自分で判定すると「操作が成功した → テストは Pass」と短絡する。特に、期待結果を知った状態で操作すると、期待に合うような報告をするバイアスがかかる。

だから：
- **qa-runner には期待結果を渡さない**。「やったこと」と「見えたもの」を客観的に報告するだけ
- **qa-judge は操作しない**。runner の報告と期待結果を照合し、新鮮なコンテキストで判定する

この分離により、判定の質が劇的に向上した。v1（分離前）では Inconclusive だった判定が、v2 以降は明確な Pass/Fail を出せるようになった。

### 2. 段階的開示 — コンテキストの知的管理

Claude Code のコンテキストウィンドウは有限のリソース。すべての知識を一度にロードすると、肝心な指示が埋もれる。

5つのスキルに分割したのは、フェーズごとに「今必要な知識だけ」をロードするため:

| Phase | スキル | ロードされる知識 |
|-------|--------|----------------|
| 0 | ios-qa-workflow | QA Readiness 検査の判定基準 |
| 1 | ios-qa-prepare | テストケースフォーマット仕様、依存関係解決 |
| 2 | ios-qa-execute | runner/judge パイプライン、状態管理 |
| 3 | ios-qa-report | レポートフォーマット、改善提案テンプレート |
| — | ios-qa-create | テストケース設計ガイドライン、カバレッジ分析 |

テストレポートを生成するときに、テストケースのパース仕様は不要。テストを実行するときに、レポートのフォーマットは不要。必要な知識を必要な時にだけ展開する。

### 3. コンテキスト隔離 — ログの封じ込め

`snapshot_ui` の出力は1回 500〜2,000 トークン。テスト1件で 5〜10 回呼ぶと 5,000〜20,000 トークン。5件のテストで 25,000〜100,000 トークン。これが親コンテキストに流れるとすぐに枯渇する。

qa-runner はフォアグラウンドサブエージェントとして実行され、UI 操作ログはその内部で消費される。親（ios-qa-workflow）に返るのは構造化されたサマリー（500〜1,000 トークン）だけ。

## 意図ベーステストとは何か

テストケースは「操作手順」ではなく「意図」を記述する。

```
❌ 悪い例（操作手順）:
1. 画面右上の + ボタンをタップ
2. セッション作成シートの「セッションを作成」をタップ
3. テキストフィールドをタップしてフォーカス
4. 「テスト」と入力
5. 送信ボタンをタップ

✅ 良い例（意図）:
新規セッションを作成し、メッセージを送信する。
```

AI が「どうやって」を自分で考える。App Map に画面構造の知識があれば効率的に、なければ探索的に。UI が変わっても「新規セッションを作成し、メッセージを送信する」という意図は変わらないので、テストケースの修正は不要。

## 保守的判定の原則

テストにおいて最も危険なのは「偽 Pass」— 壊れているのに通ってしまうこと。偽 Fail（動いているのに失敗）は人間が確認すれば済む。偽 Pass は気づかないまま本番に出る。

だから qa-judge は保守的に判定する:
- 迷ったら **Inconclusive**（Pass でも Fail でもない「判定不能」）
- Confidence が low の Pass は Inconclusive と同等に扱う
- すべての期待項目に明確な証拠がある場合のみ Pass

## なぜテスト結果を App Map に含めないか

App Map は「UI の構造と操作方法」に関するナレッジベース。テスト結果（成功率、失敗パターン等）を含めると、runner や judge に確認バイアスを与えるリスクがある。

「この画面のテストは前回 Pass だった」という情報が runner に見えると、「今回も多分大丈夫だろう」と探索が甘くなる。judge が見ると「前回 Pass だから今回も Pass」と判定が甘くなる。

App Map には**事実**だけを記録する:
- この画面にはこのボタンがある
- このボタンは tap(label:) で操作できる（信頼度 HIGH）
- この遷移は tap "Compose" で実現できる

「良い」「悪い」「成功した」「失敗した」という価値判断は App Map に入れない。
